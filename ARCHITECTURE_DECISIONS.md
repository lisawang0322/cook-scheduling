# Hot Food Cook Scheduling System — Architectural Decisions Memo

**Project:** AI-Powered Hot Food Cook Sequencing System  
**Date:** June 13, 2026  
**Version:** v2.2 (Synthetic-Data Prototype)  

*Note: All features, evaluations, and dashboards are trained and evaluated on 6 months of simulated, synthetic store data (seed=42). Production data integration is a future phase.*

---

## 1. The Five-Layer Stack

**Layer 1: Data Layer → Build** (Synthetic data generator) — Need domain-specific constraints (LCU, hold times, equipment parallelism) that off-the-shelf data doesn't provide.

**Layer 2: Feature Engineering → Build** (Custom pipeline) — Historical waste patterns, pairwise comparisons, and temporal aggregates are unique to this problem.

**Layer 3: Model → Build** (GradientBoosting) — Tabular ranking task, not generative; scikit-learn is enterprise-ready and CPU-efficient for store deployment.

**Layer 4: Inference → Build** (Local CPU) — Store terminals lack GPUs; 10-20ms latency meets requirements without cloud dependency.

**Layer 5: UI → Build** (Streamlit prototype) → Buy (Production terminal integration) — Prototype for demos; production will integrate with existing Food Planner terminal.

### Knowledge Sources
- **Cook logs:** ~23,000 synthetic events (6 months, seed=42)
- **POS sales:** ~164,000 simulated transaction records
- **Write-off logs:** ~19,600 entries with simulated quality issues

### Scope
- All data is synthetic, modeling 3 store types (urban/suburban/highway) over 6 months
- Private data (simulated convenience-store operations), not shared across tenants

### Isolation
- **Temporal split:** Training partition (Days 1-120, Jan-Apr) vs. Testing partition (Days 121-180, May-Jun)
- **Leakage prevention:** Historical feature aggregates (e.g., avg waste by hour) computed ONLY from training partition

### Refresh Cadence
- **Current prototype:** Static synthetic dataset (no refresh)
- **Future production:** Daily retraining with new POS/write-off data

## 3. Model + Fine-Tune Posture

### Tier Choice
**Tier 2 (GradientBoosting on CPU)** — Optimal balance of accuracy (74.3%), interpretability, and store-deployment feasibility without GPU requirements.

### 12-Month Fine-Tune Roadmap

**Months 1-3: Production Deployment & Monitoring**
- Deploy v2.2 as-is to pilot stores
- Monitor accuracy drift and feature importance stability
- Establish baseline override rate and associate adoption metrics

**Months 4-6: First Fine-Tune on Real Data**
- Fine-tune on first 3 months of live store outcomes
- Target: +2pp accuracy lift (from 74.3% to 76%+)
- Validate that synthetic-trained patterns transfer to real data

**Months 7-9: Distillation Evaluation (Conditional)**
- Evaluate need for distillation if hardware constraints emerge in fleet rollout
- If latency <5ms required, distill to smaller model (single decision tree or logistic regression)
- Otherwise, maintain current GradientBoosting model

**Months 10-12: Advanced Ensemble or Feature Expansion**
- If accuracy plateaus below 80%, consider ensemble methods (v2.2 + v1 weighted)
- Explore additional features: real-time sales velocity, weather patterns, local events
- Target: 80%+ accuracy for full fleet deployment

### Labeling Methodology and Benchmark Interpretation

**Deterministic label generation:** Ground-truth `optimal_order` labels are generated by a deterministic scoring formula in `src/data_labeler.py` (`_determine_optimal_order()`), which combines urgency (1/time_remaining), demand density (demand/LCU), hold-time penalty, and historical waste outcomes into a composite priority score. This formula encodes domain-expert logic for cook sequencing.

**Benchmark interpretation:** Because labels derive from a deterministic formula rather than observed real-world outcomes, accuracy metrics (including the 74.3% test-set top-1 accuracy for v2.2) measure how well each approach (v1 rules, v2.2 ML, LLM) recovers the domain-expert labeling logic, not real-world optimality. This is a strength for robustness validation: an independent LLM converging on similar rankings validates that the labels encode learnable domain patterns rather than arbitrary scoring. Future evaluation on real POS/write-off data will be required to confirm real-world waste reduction.

## 4. Evaluation Strategy

Two complementary eval sets are maintained. **Never substitute one for the other when citing authoritative model scores.**

| Eval set | Examples | Purpose | Canonical for |
|---|---|---|---|
| **ML holdout** (temporal split ≥ 2025-05-01) | 1,747 scenarios | Training / test split for v1 and v2.2 | Reporting v1 and v2.2 accuracy in isolation |
| **LLM shared eval** (`data/llm_eval_set_v0.1.json`) | 50 curated examples | Side-by-side comparison: floor → ML → LLM ceiling | Cross-model comparisons; all four comparators scored on identical inputs |

### Canonical numbers (as of Sprint 1)

| Comparator | Score | Context |
|---|---|---|
| associate_floor | 42.0% top-1 | 50-example shared eval |
| v1 rules | **71.8%** top-1 (6,480 pts) | ML holdout (authoritative) |
| v2.2 ML | **74.3%** top-1 (1,747 scenarios) | ML holdout, temporal split (authoritative) |
| llm_v01_zero_shot | 36.0% top-1 | 50-example shared eval |

### Data leakage note
`synthetic_logs` examples in the LLM shared eval (20/50) are drawn from the **training partition** (pre-2025-05-01). v2.2 has already seen these during training, so its 72.0% score on the shared eval is **not** a clean holdout result — use the 74.3% ML holdout number for authoritative v2.2 reporting.

### Where each number lives
- ML holdout numbers: `output/v1_eval_report.json`, `output/v2_2_temporal_report.json`
- Shared eval (all four comparators): `output/llm_eval_v0.1_report.json` — the `canonical_holdout_reference` block embeds the ML holdout numbers for single-artifact convenience

## 5. Acceptance Criteria

### Quality Criterion (Accuracy)
For a decision point with multiple overlapping daypart windows, the system generates a ranked cook queue with ≥ +15.0 percentage points accuracy lift over the simulated associate baseline (55.0%) and maintains a temporal holdout test within 5% of cross-validation performance, measured by top-1 ranking correctness on the Days 121-180 validation partition.

### Latency Criterion
For a single decision point scenario with 4 oven items, the system computes and returns the ranked recommendation queue with ≤100ms end-to-end latency on a standard CPU (no GPU) and processes batch evaluation of 1,000 scenarios in under 10 seconds, measured by wall-clock inference time on a back-office PC specification.

### Failure-Mode Criterion
For a decision point where the ML model's prediction confidence falls below 70% or encounters missing/invalid input features, the system automatically falls back to the deterministic v1 rule-based priority scoring (urgency × demand_density × waste_penalty) and surfaces a clear fallback indicator in the UI, measured by successful fallback execution rate in edge-case stress testing (equipment failures, extreme demand spikes, data gaps).

## 5. Sprint 1 Backlog

Sprint 1 delivers the core recommendation experience as 6 shippable features, each a vertical slice (model → logic → UI) demonstrable on synthetic data.

### Feature 1: Cook Sequence Recommendation Queue (High, 8 pts) — [COMPLETED]
**As a** store associate at the hot bar, **I want** a ranked "Cook Now / Next / Then" queue when multiple items are due, **so that** I can make one decision and start cooking without guessing.

**Acceptance Criteria:**
- Ranks pending oven items (pizza, wings_2h, wings_4h, baked_goods) for any decision point
- Surfaces v2.2 model ranking with v1 rule-based fallback
- Displays per-item cook time, hold time, ready time, and expiry
- Taquitos shown as parallel roller-grill task, not in oven queue
- Color-coded queue (red = now, yellow = next, green = then)

### Feature 2: Plain-Language Recommendation Explanations (High, 3 pts) — [COMPLETED]
**As a** store associate with minimal food training, **I want** a short reason for each recommendation, **so that** I trust the system and understand why an item is first.

**Acceptance Criteria:**
- Template-driven explanation for every ranked item (no LLM)
- Cites dominant factor(s): window closing soon, high demand, perishability
- Explanation generated for 100% of recommendations
- Wording validated as entry-level associate reading level

### Feature 3: Scenario Comparison View (High, 5 pts) — [COMPLETED]
**As a** business stakeholder evaluating the system, **I want** to see Associate vs. v1 vs. v2.2 side-by-side on a real scenario, **so that** I can judge where and why the ML recommendation differs and is better.

**Acceptance Criteria:**
- Select any synthetic decision scenario and view all three rankings side-by-side
- Highlights where v2.2 diverges from associate baseline
- Shows resulting waste/outcome difference per approach
- Includes plain-language explanation from Feature 2

### Feature 4: Impact Dashboard (Medium, 5 pts) — [COMPLETED]
**As a** regional operations manager, **I want** aggregate KPIs across thousands of decision points, **so that** I can see projected waste reduction by store type and time of day.

**Acceptance Criteria:**
- Accuracy lift (Associate vs. v1 vs. v2.2) across synthetic dataset
- Breakdowns by store type, hour-of-day, and item
- Projected waste-reduction view, grounded in `CostModel_lisaw2.xlsx`
- All figures labeled as synthetic-prototype projections

### Feature 5: What-If Scenario Simulator (Medium, 5 pts) — [COMPLETED]
**As a** store operations manager, **I want** to adjust store conditions and see the recommendation update live, **so that** I can test demand spikes and edge cases before trusting the system.

**Acceptance Criteria:**
- Adjustable inputs: store type, hour, per-item forecast demand, time-of-day
- Recommendation queue and explanations recompute in real time
- Supports edge cases (demand spikes, near-expiry windows)
- Shows how ranking shifts when inputs change

### Feature 6: Associate Override & Feedback Capture (Low, 3 pts) — [COMPLETED]
**As a** store associate who retains full control, **I want** to override the recommended order and have my choice recorded, **so that** the system respects my judgment and improves over time.

**Acceptance Criteria:**
- Associate can reorder or reject recommended queue (augmentation, not automation)
- Overrides logged with scenario context for future model learning
- Override rate surfaced as trust/adoption metric
- No food-safety decision ever auto-executed by the system

### Sprint Dependencies

**External Dependencies:** None required for synthetic-data feature build. Future phase: Food Planner API access credentials and POS/Inventory data pipeline approval.

**Technical Risks:** Synthetic data may not capture real-world complexity; model may overfit to synthetic patterns; stakeholder buy-in challenging without live-store validation.

**Mitigation Strategies:** Build each feature as vertical slice demonstrable on synthetic data; keep v1 rules as deterministic fallback; ground impact/ROI views in `CostModel_lisaw2.xlsx`; capture associate overrides early to seed future retraining.

## Future Improvements

### Model Explanation Layer (not yet implemented)

**Current state:** Verbal explanations shown in the Hot Food Hero UI (both v1 rule order and v2.2 ML ranking) are **rule-templated strings** — post-hoc `if/else` logic keyed on `decision_hour`, `demand`, `hold_time`, and `store_type`. They do not reflect the actual decision path of the GBM model.

**Option A — SHAP-based explanations (recommended next step)**
Use `shap.TreeExplainer(model)` on the v2.2 GradientBoosting model to compute per-feature Shapley values at inference time. Map the top contributing features (e.g. `pizza_forecast_demand`, `decision_hour`, `baked_goods_hold_time`) to human-readable phrases. This would make the explanation genuinely model-faithful rather than heuristic.

Approximate effort: add `shap` to `requirements.txt`, call `explainer.shap_values(X_row)` in `get_v22_ranking()`, return top-3 features with their direction and magnitude alongside the ranking.

**Option B — LLM post-processing**
Pass the ranked output + feature dict to an LLM (e.g. GPT-4o-mini) with a structured prompt to generate a natural-language rationale. Higher quality prose but adds latency, cost, and an external API dependency — not suitable for in-store deployment without connectivity guarantees.

**Decision:** Defer both options. They do not improve model accuracy and add engineering overhead. Revisit if associate trust/adoption metrics show explanation quality is a blocker.
